# 2026-07-23 — Auth MVP: email/password JWT with session hooks

## Context

Area Todo needs user accounts so todos and (later) area-pinned login sessions are scoped to a person. Architecture and frontend rules already assume JWT. Product context calls out area-pinned sessions (e.g. pin a desktop login to "office"), but that behavior is not required for the first auth slice.

## Decision

Ship an **email + password** MVP with **JWT access tokens** (Bearer header) and a persisted **Session** row per login/register.

- Hash passwords with **Argon2id**.
- Put `sub` (user id) and `sid` (session id) in the JWT so logout can revoke the session and a future area-pin feature can attach to the same session without changing claim shape.
- Reserve nullable Session fields (`pinned_lat`, `pinned_lng`, `pinned_label`) as hooks; auth ignores them until a dedicated area-pin spec lands.
- Defer OAuth, email verification, password reset, and refresh tokens.

Specs: [`../specs/backend/user.md`](../specs/backend/user.md), [`../specs/backend/auth.md`](../specs/backend/auth.md).

## Consequences

- Clients must re-login when the access token expires (no refresh flow yet).
- Bearer tokens require careful client storage; cookie-based delivery can be revisited later.
- Area pinning can extend Session (or replace reserved columns with an Area FK) without redesigning register/login.
- Full user profile CRUD remains a separate spec; registration is owned by auth.
